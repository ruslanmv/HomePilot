# Batch 1 — Hybrid Scene Rendering (Diffusion + Motion Graphics)

**Status:** design only — no implementation in this batch document.
**Depends on:** Batch 0 (`scene_kind` classification exists on outline entries).
**Proves:** diagram/quote/proof/CTA scenes render deterministically with exact, correct text; hero/atmosphere scenes keep the diffusion look.

This is the load-bearing decision in the whole design.

---

## 1. Why not "just use the best video model"

Wan 2.2, Hunyuan Video 1.5, LTX-Video, and Mochi 1 are all genuinely strong in 2026 — but none of them reliably render *exact*, *correct*, multi-word text: box labels, arrows that point at the right thing, a diagram that says "typed memory experts" and not something that looks like it. Qwen-Image 2.0 is a real step forward specifically on text fidelity for still images, and is worth evaluating for quote cards — but "a real step forward" is not "guaranteed correct," and the brief is explicit that the result must not look like generic AI motivational content. For an essay whose entire value is precise technical diagrams, handing the diagrams to a diffusion model is the single most likely way to end up with exactly the look this is trying to avoid.

The fix isn't a better prompt. It's routing diagram/data/quote/CTA content to a renderer that **cannot** hallucinate a label, and reserving diffusion for the shots where photographic/atmospheric texture is actually the point.

## 2. `scene_kind` → `rendererKind`

`scene_kind` (from Batch 0) maps directly to a new `rendererKind` field on the outline entry and on `StudioScene`:

```python
# backend/app/studio/models.py — additive
RendererKind = Literal["diffusion", "motion_graphic"]   # NEW

SCENE_KIND_TO_RENDERER: Dict[str, RendererKind] = {
    "hero": "diffusion", "transition": "diffusion",
    "diagram": "motion_graphic", "proof": "motion_graphic",
    "quote": "motion_graphic", "cta": "motion_graphic",
}
```

`StudioScene` gains an optional `rendererKind: Optional[RendererKind] = None` — `None` means "diffusion, exactly as today," so every existing scene row remains valid and every existing flow is untouched.

Editable per-scene in the existing scene editor — the mapping is a default, not a hard rule.

## 3. The `diffusion` path — existing infra, unchanged

Title-card hero shots, section-opener atmosphere, thumbnail source images. Uses the ComfyUI `txt2img` → optional `img2vid` pipeline exactly as it exists today (`backend/app/orchestrator.py`), just with:

- the model registration fixed in Batch 3, and
- a StyleKit-driven prompt/negative-prompt template (the `ruslanmv-essays` kit from Batch 0) so output stays on-brand instead of generic.

No orchestrator changes in this batch.

## 4. The `motion_graphic` path — new, and it's code, not a model

New dependency: **Remotion** (React-based, frame-exact, renders MP4/WebM directly, and — usefully — can import an existing video or image as a `<Video>`/`<OffthreadVideo>` source inside a composition, which matters for Batch 4's final assembly).

One composition per `scene_kind` archetype, under `frontend/src/ui/studio/remotion/`:

| Composition | Renders | Props (from structured scene content) |
|---|---|---|
| `DiagramScene` | animated node/edge diagrams | node labels, edge labels, arrow topology |
| `ProofScene` | benchmark/data callouts | exact numbers, units, source attribution |
| `QuoteScene` | thesis/quote cards | exact quote text, attribution |
| `CTAScene` | closing links/subscribe card | `EssaySource.source_links` |

Nothing here is generated by a model at inference time; it's rendered deterministically from data, the same way a chart library renders a chart. That's the entire point: **it cannot misspell a label, because there's no sampling step between the text and the pixels.**

All compositions read their colors/fonts/motion from the active `StyleKit` and lay out against the `CanvasSpec.safe_margin_pct` safe box (used properly in Batch 4's multi-aspect reflow; in Batch 1 only 16:9 is rendered).

### LLM-authored compositions

Concretely useful and current as of this year: Remotion's "Remotion Skills" agent capability (released January 2026) lets an LLM author the actual React component for a scene from a natural-language description of what the diagram should show — meaning the "LLM turns an essay section into a diagram" step doesn't require hand-building a template for every possible diagram shape. The LLM call proposes the Remotion component; it's versioned as ordinary code from there, reviewed once, and reused.

### Licensing wrinkle

Remotion is free for solo/tiny-team use, which covers Ruslan's own channel — but worth a second look if HomePilot itself ships this to a broader user base later. Tracked via `REMOTION_LICENSE_TIER=individual` in config and in the cross-batch risk list.

## 5. Net effect

Roughly half of a typical essay's scenes (the diagram/data/CTA/quote ones) render deterministically and never drift or garble; the other half (hero/atmosphere/transitions) get the actual diffusion-model texture the brief wants, without asking diffusion to do the one thing it's currently bad at.

## 6. Files touched in this batch

**New:** `frontend/src/ui/studio/remotion/` (`DiagramScene`, `ProofScene`, `QuoteScene`, `CTAScene` + a render harness)
**Modified (additive only):**
- `backend/app/studio/models.py` — `+RendererKind`, `+SCENE_KIND_TO_RENDERER`, `+StudioScene.rendererKind` (optional, default `None`)
- `backend/app/studio/routes.py` — scene-generation dispatch: `motion_graphic` scenes go to the Remotion render job instead of the orchestrator; `diffusion`/`None` scenes go through the existing path unchanged

**Config:** `REMOTION_LICENSE_TIER=individual`

## 7. Acceptance criteria

1. A scene with `rendererKind=None` or `"diffusion"` renders exactly as before this batch.
2. A `diagram` scene's rendered frame contains the exact label strings from its structured content (OCR or pixel-diff against the known-text render — no sampling drift possible by construction).
3. `rendererKind` is editable per-scene in the existing scene editor; changing it re-routes the render.
4. Topic-mode projects (no `scene_kind`) never hit the Remotion path.

## 8. Risk specific to this batch

Diffusion still occasionally drifts on text in motion_graphic-*adjacent* shots that touch diffusion output (e.g. a diffusion background behind a Remotion-rendered label). Mitigation is architectural, not a prompting fix — keep text-bearing content on the deterministic path, full stop. If a scene mixes both, the text layer is always Remotion-composited *over* the diffusion plate, never baked into the diffusion prompt.
